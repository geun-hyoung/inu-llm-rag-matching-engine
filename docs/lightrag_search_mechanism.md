# LightRAG 검색 메커니즘 분석

> 이 문서는 INU LLM RAG Matching Engine 프로젝트에서 LightRAG의 검색 과정을 분석한 내용입니다.
> 실제 KIPRIS 특허 데이터(펠티어 소자 신발 건조기)를 기반으로 작성되었습니다.

---

## 2. 데이터 삽입 과정

### 2.1 원본 데이터
```
특허 초록: "본 발명은 신발 건조기에 관한 것으로서, 보다 상세하게는
펠티어소자의 각 면에 방열부재와 냉열부재를 부착하고..."
```

### 2.2 LLM 엔티티/관계 추출

LLM이 텍스트에서 자동으로 엔티티와 관계를 추출합니다.

**추출된 엔티티 (8개)**
| 엔티티 | 설명 |
|--------|------|
| 신발 건조기 | 발명의 주체 |
| 펠티어소자 | 온도 조절 전자 부품 |
| 냉열부재 | 냉각 역할 |
| 방열부재 | 열 방출 역할 |
| 공기 순환구 | 공기 유입 통로 |
| 덕트 | 공기 통로 형성 |
| 물받이부 | 응축수 수집 |
| 단열부재 | 열 전도 차단 |

**추출된 관계 (7개)**
| 시작 엔티티 | 끝 엔티티 | 관계 |
|-------------|-----------|------|
| 신발 건조기 | 펠티어소자 | 사용 |
| 신발 건조기 | 덕트 | 포함 |
| 신발 건조기 | 물받이부 | 포함 |
| 펠티어소자 | 냉열부재 | 부착 |
| 펠티어소자 | 방열부재 | 부착 |
| 펠티어소자 | 단열부재 | 연결 |
| 냉열부재 | 공기 순환구 | 장착 |

## 3. 벡터 DB 구조

LightRAG는 엔티티와 관계를 **별도의 벡터 DB**에 저장합니다.

### 3.1 vdb_entities.json (엔티티 벡터 DB)

```json
{
  "embedding_dim": 1536,
  "data": [
    {
      "__id__": "ent-0297bf1c...",
      "entity_name": "공기 순환구",
      "content": "공기 순환구는 주변 공기가 신발 건조기로 유입되도록 하는 통로...",
      "source_id": "chunk-80cf3957...",
      "vector": [0.012, -0.034, ...]
    },
    // ... 총 8개 엔티티
  ]
}
```

### 3.2 vdb_relationships.json (관계 벡터 DB)

```json
{
  "embedding_dim": 1536,
  "data": [
    {
      "__id__": "rel-d7aceae5...",
      "src_id": "신발 건조기",
      "tgt_id": "펠티어소자",
      "content": "신발 건조기는 펠티어소자를 사용하여 습기를 제거하는 기능을 수행한다.",
      "source_id": "chunk-80cf3957...",
      "vector": [0.023, -0.041, ...]
    },
    // ... 총 7개 관계
  ]
}
```

### 3.3 핵심 포인트

- 임베딩 모델: `text-embedding-3-small` (1536차원)
- 엔티티/관계 모두 `source_id`로 원본 청크 연결
- 분리된 벡터 DB로 dual-level 검색 가능

## 4. Dual-Level 검색 메커니즘

### 4.1 키워드 추출 (Step 1)

질의에서 LLM이 두 종류의 키워드를 추출합니다.

```
질의: "펠티어 소자를 이용한 건조 기술"
                    │
                    ▼
            ┌───────────────┐
            │ LLM 키워드 추출 │
            └───────────────┘
                    │
        ┌───────────┴───────────┐
        ▼                       ▼
┌───────────────┐       ┌───────────────┐
│  Low-level    │       │  High-level   │
│  Keywords     │       │  Keywords     │
├───────────────┤       ├───────────────┤
│ - 열전 소자    │       │ - 펠티어 소자  │
│ - 온도 조절    │       │ - 건조 기술    │
│ - 냉각 효과    │       │               │
│ - 습도 감소    │       │               │
└───────────────┘       └───────────────┘
     │                        │
     ▼                        ▼
 구체적/기술적             추상적/개념적
 → 엔티티 검색용           → 관계 검색용
```

### 4.2 벡터 검색 (Step 2)

각 키워드를 임베딩하여 해당 벡터 DB에서 유사도 검색합니다.

```
Low-level: "열전 소자"                High-level: "펠티어 소자"
         │                                    │
         ▼                                    ▼
┌─────────────────────┐             ┌─────────────────────┐
│ text-embedding-     │             │ text-embedding-     │
│ 3-small 임베딩      │             │ 3-small 임베딩      │
└─────────────────────┘             └─────────────────────┘
         │                                    │
         ▼                                    ▼
    [1536차원 벡터]                      [1536차원 벡터]
         │                                    │
         ▼                                    ▼
┌─────────────────────┐             ┌─────────────────────┐
│   vdb_entities      │             │  vdb_relationships  │
│   코사인 유사도 검색  │             │   코사인 유사도 검색  │
└─────────────────────┘             └─────────────────────┘
         │                                    │
         ▼                                    ▼
   Top-K 엔티티 반환                    Top-K 관계 반환
```

### 4.3 키워드별 Top-K 전략

**중요**: 전역 Top-K가 아닌 **키워드별 Top-K**로 다양성 보장

```
키워드 1: "펠티어 소자"          키워드 2: "건조 기술"
         │                              │
         ▼                              ▼
┌─────────────────────┐       ┌─────────────────────┐
│ - 관계A: 0.99       │       │ - 관계X: 0.52       │
│ - 관계B: 0.98       │       │ - 관계Y: 0.50       │
│ - 관계C: 0.97       │       │ - 관계Z: 0.48       │
└─────────────────────┘       └─────────────────────┘
         │                              │
         ▼                              ▼
   Top-K: [A, B, C]              Top-K: [X, Y, Z]
         │                              │
         └──────────┬───────────────────┘
                    ▼
           Merge (중복 제거)
                    │
                    ▼
          [A, B, C, X, Y, Z]
```

유사도가 낮더라도 각 키워드의 결과가 포함됨!

## 6. 일반 RAG vs Graph RAG 비교

### 6.1 검색 방식 차이

| 구분 | 일반 RAG | Graph RAG (LightRAG) |
|------|----------|----------------------|
| 검색 대상 | 청크 직접 검색 | 엔티티/관계 → 청크 |
| 검색 방식 | 질의 ↔ 청크 유사도 | 그래프 탐색 |
| Multi-hop | ❌ 불가능 | ✅ 가능 |
| 간접 연관 | ❌ 못 찾음 | ✅ 그래프로 연결 |

### 6.2 Multi-hop 검색의 장점

```
질의: "펠티어 소자의 효율을 높이는 방법"

[일반 RAG]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
질의 → 청크 직접 비교
     → "펠티어 소자 효율"이 직접 언급된 청크만 찾음
     → 제한된 결과

[Graph RAG]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
질의 → 펠티어소자 (엔티티)
     → 냉열부재 (관계로 연결)
     → 공기순환구 (관계로 연결)
     → "공기순환구"가 언급된 청크도 찾아옴!

결과: 질의에 없던 "공기순환구"가 효율과 관련있음을 발견
```

### 6.3 Dual-level의 장점

| Level | 역할 | 예시 |
|-------|------|------|
| High-level (관계) | 넓은 맥락, 개념간 연결 | "건조 기술" → 전체 시스템 이해 |
| Low-level (엔티티) | 구체적 기술 정보 | "열전 소자" → 세부 부품 정보 |

두 레벨을 조합하여 **깊이 + 넓이** 모두 확보!

## 7. 검색 모드 비교

LightRAG는 4가지 검색 모드를 지원합니다.

| 모드 | 사용 DB | 특징 |
|------|---------|------|
| `naive` | 청크만 | 일반 RAG와 동일 |
| `local` | 엔티티 | Low-level 검색만 |
| `global` | 관계 | High-level 검색만 |
| `hybrid` | 엔티티 + 관계 | 둘 다 사용 (권장) |

## 8. 프로젝트 내 관련 파일

### 8.1 설정 파일
- `models/model_config.py`: 임베딩/LLM 모델 설정

- `vdb_relationships.json`: 관계 벡터 DB

### 8.3 테스트 파일 (tests/)
- `test_insert_one.py`: 단일 특허 삽입 테스트
- `test_search.py`: 검색 모드별 테스트
- `test_search_detailed.py`: 상세 검색 과정 분석
- `visualize_graph.py`: 그래프 시각화
